# 社区活动奖品层：系统设计约束清单

日期：2026-06-16

## 1. 边界约束

- 奖品层不负责活动玩法规则。
- 抽奖、签到、任务等活动类型暂不在 V1 展开。
- 审核/风控规则由活动层决定，奖品层只承接审核结果。
- 用户端奖励入口由活动层配置，奖品层只提供奖励数据和可操作动作。
- 奖品层不重新建设完整优惠券系统、完整商品库或完整营销中台。

## 2. 库存约束

- 软件码为单码单用。
- 软件码活动发放和积分商城兑换共用同一码池。
- 同一个软件码只能分配给一个用户。
- 实物商品引用商城商品并共用商城库存。
- 活动触发发奖后先占用库存，再进入审核或发放流程。
- 审核拒绝、审核超时、取消发放时必须释放已占用库存。
- 活动结束后，未生成奖励记录且未占用库存的奖品不需要清退。
- 已生成奖励记录的奖品不因活动结束自动取消。

## 3. 幂等与并发约束

- 上游活动触发发奖必须提供幂等标识。
- 同一幂等标识重复提交，不得重复生成奖励、重复占用库存或重复发放权益。
- 库存占用、奖励记录生成、权益分配之间需要保持一致。
- 软件码分配需要避免积分商城兑换和活动发放并发抢占同一个码。
- 实物商品库存占用需要避免商城购买和活动发放并发超发。

## 4. 优惠券约束

- 优惠券系统已存在，但需要支持活动奖品层按券模板发券。
- 活动支持直接到账和领取资格两种模式。
- 直接到账以优惠券进入用户账户为成功。
- 领取资格以资格生成成功为阶段成功，用户领取成功后才算券到账。
- 券模板有效期、适用范围、库存口径需要从优惠券系统暴露给活动配置。
- 发券失败支持自动重试和人工补发。

## 5. 软件码约束

- 软件码发放后允许用户查看、复制和再次查看。
- 用户只能查看自己的软件码。
- 软件码发放成功后不允许被再次分配。
- 软件码发放失败进入异常队列，支持重试和人工补发。

## 6. 实物商品约束

- 实物商品 V1 不建设独立商品库。
- 实物奖品引用商城商品主数据。
- V1 暂不要求成本、供应商、物流单号、导出发货、退换货。
- 用户填写地址后，后台维护待发货、已发货、已完成等基础状态。
- 已发货后用户不可自行修改地址。

## 7. 审核/风控约束

- 活动层决定审核/风控规则。
- 奖品层承接审核结果并执行状态流转。
- 审核通过后才能最终发放权益或进入实物履约。
- 审核拒绝后释放占用库存。
- 已发放成功的奖励不允许再被审核拒绝覆盖。

## 8. 失败处理约束

- 发奖失败同时支持自动重试和人工补发。
- 自动重试次数、间隔、终止条件待细化。
- 人工补发权限、操作记录、是否允许替换奖品待细化。
- 用户端不直接展示技术性发奖失败原因。
- 后台必须记录失败环节、失败原因、重试次数、最后处理时间和处理人。

## 9. 数据留痕约束

- 奖励记录必须保留活动、用户、奖品、来源对象、审核状态、发放状态、库存占用状态。
- 人工操作必须保留操作人、操作时间、操作类型、操作结果和备注。
- 活动结束后必须保留活动奖品配置快照。
- 奖品来源变更、商品下架、券模板变化不应影响历史奖励解释。

## 10. 权限约束

- 奖品配置、奖励记录查看、人工补发、实物履约需要区分后台权限。
- 用户只能查看和操作自己的奖励。
- 用户敏感信息和收货信息需要受后台权限控制。

## 11. 待系统设计细化项

- 自动重试策略。
- 人工补发是否允许替换奖品。
- 审核超时释放库存规则。
- 共享库存占用与释放的事务边界。
- 活动层审核结果传递协议。
- 优惠券系统按模板发券能力改造范围。
- 软件码码池和商城库存并发控制方案。
